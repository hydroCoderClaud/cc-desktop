# 会话应用当前设计与实现对齐说明

> 最后更新：2026-06-30
> 当前版本：v1.7.83
> 本文档描述当前已经落地的“会话应用”能力，重点说明它现在如何工作、哪些语义已经稳定、以及和普通 Agent 会话的复用边界。

---

## 一、定位

会话应用不是另一套独立运行时，也不是新的顶层模式。

它的本质是：

- 基于现有 Agent 会话体系沉淀出的“可复用会话定义”
- 用一组稳定字段约束某类会话的目标、启动方式和默认上下文
- 让用户能反复启动同类会话，而不是每次都重新描述工作流

因此，会话应用更接近：

**可重复启动的专用会话模板**

而不是：

- 复杂前端内嵌应用
- 独立 SDK App
- 普通一次性聊天记录

---

## 二、当前核心模型

当前实现采用单一定义模型：

- 一个应用只有一份当前定义
- 不再保留“版本 / 草稿 / 已发布”的业务语义
- 创建即生效
- 修改直接覆盖当前定义

当前稳定字段包括：

- `appId`
- `name`
- `description`
- `icon`
- `systemPrompt`
- `startupMessageTemplate`
- `defaultContext`
- `createdFromSessionId`

其中当前面向用户稳定暴露的编辑面主要是：

- 名称
- 说明
- 启动消息模板
- 系统提示词
- 默认工作目录

`allowedCapabilities`、`inputSchema`、`workflowHints`、`outputHints` 仍保留在底层模型中，但当前不作为稳定用户编辑面，不应在普通产品文档中强调。

---

## 三、它和普通会话的关系

会话应用不会替代普通会话。

二者关系是：

- 普通会话：即时、开放、一次性，用户临时提出需求即可开始
- 会话应用：把已经打磨好的目标、规则和启动方式提炼出来，变成可重复启动的专用会话入口

启动一个会话应用时，本质上仍然是创建一个新的 Agent 会话，只是这个新会话会自动带上应用定义。

当前复用方式包括：

- 共用 `AgentSessionManager`
- 共用会话数据库
- 共用左侧会话列表
- 共用聊天窗口
- 共用内置 MCP 注入链路

所以从宿主架构上看，会话应用不是旁路系统，而是现有 Agent 会话体系上的一层“定义与约束”。

---

## 四、当前已实现入口

当前会话应用支持以下入口：

### 4.1 能力管理工作台

入口：

- `能力管理 -> 会话应用`

可完成：

- 新建应用
- 编辑应用定义
- 复制应用
- 删除应用
- 查看关联会话
- 启动新应用会话

### 4.2 会话内创建卡片

入口：

- 工具栏会话应用按钮
- `/session-app` 命令

当前行为：

- 先生成一个可确认的会话应用卡片
- 用户可直接修改名称、说明、启动消息、系统提示词和默认工作目录
- 确认后可选择：
  - 仅保存到会话应用
  - 立即创建一个新的应用会话

这条链路的意义是：

- 把“自然语言提炼应用”的过程保留在聊天里完成
- 再由用户做一次显式确认，避免直接静默落库

### 4.3 内置 MCP

当前已接入 `hydrodesktop` 内置 MCP 的工具包括：

- `session_app_list`
- `session_app_get`
- `session_app_create`
- `session_app_update`
- `session_app_launch`

这意味着普通 Agent 会话中，模型已经可以直接通过内置 MCP 创建、查看、修改和启动会话应用，而不一定非要跳回可视化工作台。

---

## 五、启动时真实加载的内容

当前会话应用启动时，不是只加载一个名字，也不是只显示一个标签。

新会话会复用应用当前定义，并带入以下内容：

- `sessionAppId`
- 应用当前 `systemPrompt`
- 应用当前 `allowedCapabilities` 对应的能力边界
- 应用当前 `defaultContext`
- 应用当前 `startupMessageTemplate`

其中最直接影响用户体感的是两项：

### 5.1 系统提示词

系统提示词不会直接显示在聊天区，但会参与该应用会话的系统上下文构建。

它适合承载：

- 目标限定
- 角色边界
- 规则约束
- 推荐使用哪些 skill / CLI / MCP
- 不应做什么

### 5.2 启动消息模板

启动消息模板会在新建应用会话后，自动作为首条用户消息发送给 Agent。

它适合承载：

- 每次启动时都要执行的固定开场任务
- 工作输入模板
- 对当前任务的明确触发语句

因此：

- `systemPrompt` 更像隐藏的长期规则
- `startupMessageTemplate` 更像每次启动时自动发出的开场白

---

## 六、工作目录语义

当前会话应用采用两层工作目录语义：

### 6.1 应用定义里保存的是基础工作目录

应用定义中的 `defaultContext.cwd` 表示该应用的默认基础目录。

新建应用时：

- 用户可手工指定
- 如果不指定，则默认使用全局 Agent 输出目录下的 `sessionapp`

也就是：

- Windows 示例：`C:/Users/xxx/cc-desktop-agent-output/sessionapp`
- macOS 示例：`/Users/xxx/cc-desktop-agent-output/sessionapp`

### 6.2 每次启动新会话时，会在基础目录下再创建随机子目录

启动应用会话时，不会直接把基础目录拿来作为本次会话目录，而是会在其下追加一个随机会话子目录，例如：

- `.../sessionapp/conv-a1b2c3d4`

这样做的目的是：

- 保证每次新应用会话有独立工作空间
- 避免不同运行实例互相覆盖输出
- 保持“应用定义目录”和“具体运行目录”分层

---

## 七、历史会话与应用定义的关系

当前设计中，历史会话不会保存一份独立的应用定义快照。

真实行为是：

- 会话只保存 `sessionAppId` 及必要的会话绑定信息
- 当该会话再次运行、继续发送消息、或重建 query options 时，会读取该应用的当前定义

这意味着：

- 应用修改后，新启动会话会使用最新定义
- 历史应用会话再次继续使用时，也会跟随当前定义，而不是锁死在旧快照

当前刻意不保留隐藏快照，也不保留内部应用指纹，原因是：

- 产品希望保持模型简单
- 避免引入“为什么这个历史会话还是旧定义”的额外心智负担

代价是：

- 应用定义修改后，历史会话继续运行时也会受影响

这是当前有意选择的简化方案。

---

## 八、删除语义

删除会话应用时，当前行为是：

- 删除应用定义
- 不删除已经存在的历史会话
- 历史会话保留原会话记录

但删除后：

- 这些历史会话不再能继续作为“某个仍存在的应用”被管理
- 运行中的内存会话会移除 `sessionAppId` / `sessionAppInput`
- 从用户视角看，历史会话会逐步回到普通会话语义

当前删除弹窗会在存在关联会话时给出慎重提示。

---

## 九、和复杂内嵌 App 的边界

会话应用当前明确不是复杂内嵌 App 方案。

适合会话应用的，是：

- 会话/流程型
- 主要承载目标、规则和固定开场流程
- 界面需求弱
- 重点是反复复用同类工作流

不适合直接交给会话应用的，是：

- 强交互复杂界面
- 大量前端状态机
- 重业务数据面板
- 强图形化操作流程

这类能力仍更适合：

- 由团队自己开发复杂内嵌 App
- 或未来走 Hosted SDK / 独立 App 方向

---

## 十、当前 macOS 兼容性结论

本轮会话应用实现对 macOS 的核心兼容性结论如下：

- 默认目录构造使用 Node `path.join`，主进程可正确生成 POSIX 路径
- 渲染层默认目录展示统一用 `/` 展示，不依赖 Windows 盘符
- 启动新应用会话时的随机子目录拼接对 POSIX 路径有效
- 文件夹选择走统一 `dialog:selectFolder`，不依赖 Windows 专属对话框语义

当前未发现会话应用主链路存在必须立刻修复的 Windows-only 假设。

本轮重点补的是：

- 补充 POSIX/macOS 路径语义测试
- 在文档中明确默认目录与新会话子目录行为

---

## 十一、当前最重要的产品结论

当前会话应用已经形成稳定定义：

1. 它是会话级应用，而不是复杂 UI App。
2. 它复用 Agent 会话体系，而不是另起一套架构。
3. 它适合承载“固定目标 + 固定规则 + 固定启动方式”的可重复工作流。
4. 它支持在工作台中管理，也支持在普通聊天中自然语言创建和修改。
5. 它已经接入内置 MCP，因此可以成为桌面“自我扩展”的第一层稳定能力。

如果后续继续扩展，最合理的方向不是把它做成“复杂内嵌 App 编辑器”，而是继续把它打磨成：

**对普通用户最友好的轻量会话应用层**

如需继续推进“运行后分析、给出升级建议、由用户确认写回”的后续能力，见：

- [session-app-optimizer-design.md](./session-app-optimizer-design.md)
