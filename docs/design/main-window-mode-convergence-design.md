# 主窗口模式收口与工作台归位设计

> 最后更新：2026-06-29
> 当前版本：v1.7.80
> 本文档描述主窗口产品定位与信息架构调整方向，仅定义产品与架构边界，不包含实施拆解与代码改造方案。

---

## 一、背景

`cc-desktop` 发展过程中，主窗口逐步承载了三类顶层形态：

- Developer
- Agent
- Notebook

这三者在实现上形成了并列“模式”，但在产品定位上并不对等：

- `Agent` 更接近主产品本体
- `Developer` 更接近内部能力与历史运行壳
- `Notebook` 更接近带强业务语义的专业工作台

随着产品方向逐步收敛到“面向企业内部的个人 AI 桌面”，继续对外强调多模式，会带来以下问题：

1. 用户心智被模式切换分散，主产品定义不清晰。
2. 新能力容易继续以“新增模式”方式进入主窗口，导致结构继续膨胀。
3. `Notebook` 这类专业工作台与真正的 embedded app 体系边界不清，影响后续扩展治理。
4. `Developer` 作为公开入口会持续强化“平台”而非“产品”的感知。

因此需要先完成一次设计层面的收口：不立即重构实现，但先统一产品叙事、信息架构和未来扩展规则。

---

## 二、设计目标

本轮设计目标仅包含以下四项：

1. 对外统一 `cc-desktop` 的主产品定位。
2. 收敛主窗口“多模式”心智，避免继续扩散。
3. 在不大改现有实现的前提下，为 `Notebook` 和未来专业工作台明确归类。
4. 为未来新增能力建立稳定入口规则，避免再次引入新的顶层模式。

---

## 三、非目标

本设计明确不包含以下内容：

- 不要求立即删除 Developer 相关代码与运行链路
- 不要求立即将 Notebook 迁移到 embedded app 注册表体系
- 不要求重写主窗口导航、会话状态或左右面板结构
- 不要求在本轮统一所有历史窗口与 app 类型
- 不要求输出实施任务拆解、排期或代码改造清单

本设计优先解决“产品定位与结构边界”，不在此阶段推动高风险实现调整。

---

## 四、设计结论

### 4.1 主产品定位

`cc-desktop` 对外统一定义为：

**面向企业内部的个人 AI 桌面**

该定义下，主窗口应被理解为单一产品壳，而不是多个并列产品模式的集合。

### 4.2 主窗口心智

主窗口不再继续强化 “Developer / Agent / Notebook” 三个并列模式的产品表达。

设计上应逐步转向以下结构：

- 主产品：AI 桌面
- 默认主工作区：Agent 对话与日常协作
- 专业工作区：Notebook 等工作台
- 内部能力入口：Developer

也就是说，未来主窗口只有一个对外产品心智，不再鼓励用户先理解“当前处于哪个模式”。

### 4.3 Notebook 归类

`Notebook` 在产品定义上不再视为与 Agent 并列的正式模式，而应归类为：

**AI 桌面内的专业工作台 / 特殊类型工作区**

它与 embedded app 的目标方向一致，但本轮仅完成“概念归位”，不强制要求实现立即并轨。

### 4.4 Developer 归类

`Developer` 不再作为主产品公开叙事的一部分，而应归类为：

**内部保留的高级能力入口 / 运行壳**

它可以继续存在于系统中，但不应继续作为普通用户的一线入口与公开定位。

### 4.5 未来扩展规则

从本设计生效后，新增专业能力不再以“新增顶层模式”的方式进入主窗口。

统一规则为：

- 历史遗留的 `Notebook` 暂保留特殊处理
- 未来新增工作台、专业场景页面、行业化能力，一律按 embedded app 方向设计

这条规则是本设计最重要的治理约束之一。

---

## 五、信息架构调整方向

### 5.1 调整后的概念分层

建议采用如下分层：

```text
AI Desktop（主产品）
  ├─ 主工作区（Agent）
  ├─ 专业工作台（Notebook 等）
  ├─ 内嵌 App（未来新增能力）
  └─ 内部高级入口（Developer）
```

其中：

- `Agent` 是默认主工作区
- `Notebook` 是历史形成的专业工作台
- `Embedded App` 是未来专业能力的标准进入方式
- `Developer` 是内部保留能力，不再承担对外产品定位

### 5.2 Notebook 的过渡定位

`Notebook` 当前虽然在实现上仍然占用 `AppMode.NOTEBOOK`，但设计上应按以下原则理解：

1. 它不是未来新增模式的模板。
2. 它是历史遗留的特殊工作台。
3. 它的产品表达应尽量向“打开一个工作台”靠拢，而不是“切换到另一种产品模式”。
4. 当未来架构演进条件成熟时，可再评估是否并入 embedded app 标准体系。

也就是说，本设计允许 `Notebook` 在一段时间内保持“概念上像 app、实现上仍是 mode”的过渡状态。

### 5.3 Developer 的过渡定位

`Developer` 当前仍承载终端、多会话、插件、Skills、MCP 等历史能力。

本设计不否认这些能力的价值，但强调其入口属性需要降级：

1. 对普通用户隐藏或弱化。
2. 对内部研发与调试保留。
3. 不再作为对外版本的主卖点之一。
4. 后续如需继续演进，应以“高级能力入口”而非“平行产品模式”来定义。

## 六、设计原则

### 6.1 先收口产品，再清理架构

本轮优先级不是架构纯化，而是产品表面收敛。

原因是：

- 当前实现中，主窗口、会话、左右面板、设置同步、Notebook 工作台均已建立稳定耦合
- 若直接在本轮强推模式重构，风险显著高于收益
- 产品叙事可以先变正确，代码可后续渐进清理

因此，本设计明确接受一段时间内的“概念先行、实现滞后”。

### 6.2 只允许一个对外主心智

主产品只能有一个一线叙事，即“企业内部的个人 AI 桌面”。

其他形态只能是：

- 工作区
- 工作台
- 内嵌 app
- 内部入口

不能再次回到“多个并列主模式”的结构。

### 6.3 新增能力必须走标准化入口

未来新增能力必须有清晰归类，不再允许临时增加新的顶层 mode。

这条约束用于防止历史问题继续扩散。

### 6.4 过渡期允许历史例外，但不允许新增例外

`Notebook` 可以作为过渡期特例保留。

但从本设计开始，不应再出现第二个以历史理由直接进入顶层 mode 的新能力。

---

## 七、建议的对外表达

### 7.1 产品层表达

对外统一使用以下表达方式：

- `cc-desktop` 是企业内部个人 AI 桌面
- Notebook 是其中的专业工作台
- 其他专业能力未来会以工作台或内嵌 app 形式进入

避免继续使用以下表达：

- 三种主模式并列
- Developer 是普通用户主要入口之一
- Notebook 是与 Agent 平级的长期模式体系

### 7.2 文案层表达

后续产品、文档、演示、介绍材料应逐步朝以下语言风格收口：

- “打开 Notebook 工作台”
- “进入专业工作区”
- “在 AI 桌面中使用某个内嵌应用”
- “高级能力入口”或“内部调试入口”

不再优先使用：

- “切换模式”
- “开发者模式 / 智能体模式 / Notebook 模式”并列讲述

---

## 八、风险与取舍

### 8.1 接受的取舍

本设计接受以下现实：

1. 设计语义会先于代码结构收敛。
2. 一段时间内，文案与实现命名可能并不完全一致。
3. `Notebook` 会保持一段时间的特殊地位。
4. `Developer` 隐藏后仍保留历史实现和内部价值。

这些取舍是为了换取较低的变更风险和较高的产品收口效率。

### 8.2 主动避免的风险

本设计希望主动避免以下风险：

1. 为了追求概念纯净而过早重构主窗口
2. 把 Notebook 强行塞入尚未完全匹配的 embedded app 机制
3. 删除 Developer 导致内部工作流与调试能力回退
4. 在未完成定位收口前继续引入新的模式型入口

---

## 九、后续演进边界

本设计之后，如需继续演进，可按以下顺序思考：

1. 先验证“单一主产品 + 工作台”叙事是否稳定
2. 再评估 Notebook 是否适合逐步并轨到 embedded app 体系
3. 再评估 Developer 是否需要进一步收缩为高级工具入口
4. 最后再考虑是否清理历史 `AppMode` 结构

换言之，后续任何实现改造都应服务于本设计结论，而不是重新打开“是否继续保留多模式产品”的讨论。

---

## 十、最终结论

本设计的最终结论如下：

1. `cc-desktop` 对外统一为“面向企业内部的个人 AI 桌面”。
2. `Agent` 是默认主工作区，不再与其他形态并列为多个主产品模式。
3. `Notebook` 在设计上归位为专业工作台，但本轮不强制迁移实现。
4. `Developer` 在设计上归位为内部高级入口，但本轮不强制删除实现。
5. 未来新增专业能力一律按 embedded app 方向进入，不再新增顶层模式。

该方案的核心价值不是“立刻让架构变纯”，而是“先让产品变清晰，并为后续演进建立纪律”。
