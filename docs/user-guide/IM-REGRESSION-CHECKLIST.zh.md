# IM 三端统一回归测试清单

本文档用于手工回归当前三条主验证通道：

- 飞书
- 钉钉
- 企业微信

个人微信保留现有链路，但不属于这轮“三端统一”主验收范围。

## 一、测试前准备

### 通用环境

- Hydro Desktop 已启动
- Claude Code CLI 与 API 配置正常
- 当前桌面端可正常发起 Agent 会话
- 准备一组测试附件：图片、PDF、Word、Excel、PowerPoint
- 文档附件建议覆盖普通文件名和包含空格 / 中文 / `#` / `?` 的文件名，便于验证 macOS 与特殊字符路径

### 飞书

- 飞书应用已启用
- 单聊与群聊测试环境可用

### 钉钉

- 钉钉应用已启用并发布
- Stream 模式连接正常
- `robotCode` 已配置

### 企业微信

- 智能机器人已创建并启用
- Bot ID / Secret 已配置
- 如需主动发送成员，`wecom-cli` 已完成初始化

## 二、统一主清单

### A. IM 入站首次建会话

1. 三端单聊各发一条文本消息
   - 预期：桌面端各自创建会话并写入消息
2. 三端群聊各发一条文本消息
   - 预期：桌面端按群维度建会话，不串到某个单聊用户

### B. 文本往返闭环

1. 在三端分别发送普通文本
   - 预期：桌面端收到，Agent 正常回复
2. 连续追问两轮
   - 预期：保持在同一会话，不额外新建

### C. 图片到桌面

1. 三端分别发送图片
   - 预期：桌面端正确收到
2. 飞书 / 企业微信发送图文混排
   - 预期：文本与图片都能进入会话

### D. 桌面图片回 IM

1. 在桌面端触发读取本地图片
   - 预期：三端能收到图片回传

### E. 文档附件到桌面

1. 三端分别发送 PDF
   - 预期：桌面端收到文档卡片，文件保存到当前会话目录的 `im_attachments/`
   - 预期：右侧预览可直接打开 PDF，不需要弹出保存对话框
2. 三端分别发送 Word / Excel / PowerPoint
   - 预期：桌面端收到文档卡片，Office 预览复用 Notebook 预览能力
3. 在会话中询问 Agent “读取这个文件并总结”
   - 预期：Agent 能看到本地文件路径，并按自身文件/PDF能力读取；IM 层不应提前抽取正文

### F. 桌面文档回 IM

1. 从桌面端工具栏向三端目标分别发送 PDF
   - 预期：IM 端收到原生文件消息
2. 从桌面端工具栏向三端目标分别发送 Word / Excel / PowerPoint
   - 预期：IM 端收到原生文件消息，文件名和类型正确
3. 尝试发送不支持的文件类型，例如 `.zip`
   - 预期：桌面端明确提示暂不支持，不应发出消息

### G. 会话内 MCP 发送文档

1. 在已绑定 IM 目标的会话中，让 Agent 发送本地 PDF，例如“把 `/Users/demo/report.pdf` 发给当前飞书联系人”
   - 预期：Agent 调用 `mcp__hydrodesktop__im_send`，参数包含 `filePaths`
   - 预期：目标 IM 收到文件
2. 指定目标发送，例如飞书 / 钉钉 / 企业微信各选一个明确联系人或群名
   - 预期：发送到指定目标，不串到当前会话绑定的其他目标
3. 在 Windows 和 macOS 分别测试绝对路径
   - 预期：`C:\...`、`D:\...`、`/Users/...`、`/tmp/...` 均可被识别为本地绝对路径

### H. macOS / 特殊字符预览

1. 在 macOS 上接收或选择文件名包含空格、中文、`#`、`?` 的 PDF
   - 预期：右侧 PDF 预览正常加载
2. 在 macOS 上接收或选择同类文件名的 HTML
   - 预期：右侧 HTML webview 预览正常加载
3. 使用不含特殊字符的普通 PDF 作为对照
   - 预期：两类文件预览行为一致

### I. 历史与恢复

1. IM 发起一个会话并聊一轮
2. 从桌面端关闭该会话
3. 再次从同一 IM 对话发消息
   - 预期：进入历史选择 / 新建逻辑，而不是静默串错会话
4. 选择历史恢复
   - 预期：恢复成功，后续继续消息不再重复弹菜单
5. 选择 `0`
   - 预期：创建全新会话

### J. 命令链路

1. `/help`
   - 预期：返回帮助文本
2. `/status`
   - 预期：返回历史会话状态列表
3. `/resume`
   - 预期：返回历史会话选择菜单
4. `/resume N`
   - 预期：恢复指定会话
5. `/new`
   - 预期：新建会话
6. `/close`
   - 预期：关闭当前会话
7. `/close 1`
   - 预期：明确提示不支持带编号

### K. `/status` 标记语义

检查历史列表中的三种标记：

- `✅` 当前映射会话
- `🔵` 已激活但不是当前映射
- `⭕` 历史存在但当前未激活

重点验证：

- 主动绑定但尚未成为当前映射的会话，不应错误显示为 `✅`
- 当前聊天上下文真正使用的会话，才应显示 `✅`

### L. 桌面主动发送绑定

1. 从桌面端工具栏向一个 IM 目标主动发消息
   - 预期：发送成功并建立绑定
2. 对方从 IM 回复
   - 预期：直接回到当前绑定会话
3. 从桌面端关闭该会话，再回复
   - 预期：进入历史选择 / 新建逻辑

### M. 解绑防复活

1. 桌面端主动绑定后执行解绑
2. 检查工具栏与会话图标恢复未绑定状态
3. 对方再次发消息
   - 预期：不会因为旧 runtime map 或旧 DB 关系错误复活

### N. 桌面介入

1. 在 IM 发起的桌面会话中，由桌面端继续发言
   - 预期：回复只回当前绑定的同一 IM 目标
2. 不同渠道并存时
   - 预期：不会误回到其他 IM 渠道

## 三、飞书专项

1. 群聊 `@机器人` 后发送正文
   - 预期：正文不残留机器人 mention
2. 群聊中同时 `@机器人` 与 `@其他用户`
   - 预期：送给模型的正文不保留这些 mention 噪音
3. `/help` `/status`
   - 预期：走文本 / Markdown 风格回复，而不是交互卡片

## 四、钉钉专项

1. 单聊主动绑定后，对方回复
   - 预期：直接回当前会话
2. 群聊主动绑定后，对方回复
   - 预期：按群维度回当前会话
3. 群聊再次 `/new` 或回复 `0`
   - 预期：新会话仍绑定到群，不落到单聊用户

## 五、企业微信专项

1. 回复是否原生流式返回
   - 预期：逐段输出
2. 群聊 status / resume
   - 预期：按群维度工作，不把 `@机器人` 当正文
3. 解绑后图标恢复
   - 预期：不需要重启也能恢复当前 UI 状态

## 六、当前已知非缺陷项

以下现象当前不按 bug 处理：

- 飞书、钉钉普通 Q&A 默认不是原生流式逐段返回
- 企业微信保留原生流式回复能力
- 个人微信不属于本轮三端统一主验收对象
- 企业微信主动发送依赖 `wecom-cli`
- 文档附件只支持 PDF、Word、Excel、PowerPoint；音频、视频仍不属于当前阶段
- IM 层不抽取 PDF/Office 正文；需要分析内容时由 Agent 使用文件/PDF能力读取本地路径
